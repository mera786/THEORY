7. Team Skill Requirement : Teams need to understand Spring Boot, Spring Cloud, Docker, Kubernetes, CI/CD, monitoring, discovery, security

(15). Fault Tolerance (Resilience4j, Hystrix, Circuit Breaker) :
###################################################################

Q1. What is fault tolerance?
ðŸ‘‰ it is a concept/principle , which define Ability of the system to continue working even if some services fail.

Q2. What is a Circuit Breaker pattern?
ðŸ‘‰ It's design pattern, It stops sending requests to a service that is failing for some time, so that the problem does not spread to other services.

Q3. Example of a fault tolerance library?
ðŸ‘‰ Resilience4j, Hystrix (now deprecated).

Q4. What is the difference between Retry and Circuit Breaker?
ðŸ‘‰ Retry: attempts the request again if it fails.
ðŸ‘‰ Circuit Breaker: stops making requests if failures exceed threshold.

Q5. How do you implement fallback in case a service is down?
ðŸ‘‰ Using Resilience4jâ€™s @CircuitBreaker annotation with a fallback method to return a default response.

Q6. How do you handle rate limiting in microservices?
ðŸ‘‰ With Resilience4jâ€™s rate limiter or at API Gateway level.

Q7. Can you explain a real issue where fault tolerance saved your system?
ðŸ‘‰ Example: When our Payment Service went down, Order Serviceâ€™s circuit breaker returned a fallback (â€œpayment unavailableâ€) instead of crashing the system.

Q8. What is the difference between Hystrix and Resilience4j?
ðŸ‘‰ Hystrix (Netflix) is older and now deprecated. Resilience4j is lightweight, modular, supports Java 8+ functional programming, and integrates well with Spring Boot 2+.

Q9. What is Bulkhead pattern?
ðŸ‘‰ The Bulkhead pattern is like dividing a ship into separate compartments. If one part gets damaged, water doesnâ€™t sink the whole ship.
In software, it means separating resources (like threads, memory, or connection pools) for each service/function. This way, if one service fails or gets overloaded, it wonâ€™t use up all the resources and stop other services from working.
Example: If Notification Service crashes, it wonâ€™t block resources needed by Payment Service â€” so payments can still work fine.

Q10. What is the Timeout pattern and why is it important?
ðŸ‘‰ Timeout makes sure a request to a service doesnâ€™t wait forever. If the service doesnâ€™t respond within X seconds, the call is stopped.
 Real-time example: We set a 2-second timeout for Payment Service calls, so Order Service doesnâ€™t get stuck waiting.

Q11. In terms of fault tolerance, what is the difference between Fail-Fast and Fail-Safe?
ðŸ‘‰ Fail-Fast: Stop immediately when a failure happens (e.g., Circuit Breaker).
ðŸ‘‰ Fail-Safe: Continue execution by returning a default value or fallback (e.g., Fallback method).

Q12. How do you monitor Circuit Breakers in production?
ðŸ‘‰ Using Micrometer + Prometheus + Grafana dashboards, or by exposing metrics via Actuator endpoints.

Q13. What is the Saga Pattern in microservices?
ðŸ‘‰ Saga Pattern is a design pattern used in microservices to handle transactions.
Instead of having one big transaction across many services, it breaks it into smaller local transactions.
If one of the steps fails, it uses compensation steps (undo actions) to keep the system consistent.

Q14. What are the two ways to implement Saga?
ðŸ‘‰ Choreography (services communicate via events)
ðŸ‘‰ Orchestration (a central Saga orchestrator controls the flow).

Q15. Can you give a real example of Saga?
ðŸ‘‰ In e-commerce: Order â†’ Payment â†’ Inventory â†’ Shipping. If inventory fails, trigger compensation (refund + cancel order).

Q16. Difference between 2PC (Two-Phase Commit) and Saga?
ðŸ‘‰ 2PC tries to lock all resources â†’ bad for scalability.
ðŸ‘‰ Saga is async, uses compensation instead of locking â†’ better for microservices.

Q17. How does Saga help in fault tolerance?
ðŸ‘‰ Even if one service fails, Saga ensures system consistency by rolling back earlier steps.

Q18. Why do we need a Circuit Breaker if we already have Saga?
ðŸ‘‰ Saga ensures data consistency across services.
ðŸ‘‰ Circuit Breaker prevents cascading failures when a service is down.
ðŸ‘‰ Both solve different fault tolerance problems.

Q19. How do you decide between Retry, Circuit Breaker, and Saga?
ðŸ‘‰ Retry â†’ when failure is temporary (like network glitch).
ðŸ‘‰ Circuit Breaker â†’ when service is unhealthy, stop sending requests.
ðŸ‘‰ Saga â†’ when multiple services form a business transaction.

Q20. What tools can you use to implement Saga in Spring Boot?
ðŸ‘‰ - Axon Framework, Camunda, Eventuate Tram Saga, or custom using Kafka/RabbitMQ + Choreography.

Q21. Can Saga work with Resilience4j/Hystrix?
ðŸ‘‰ Yes, you can use Circuit Breaker + Saga together:
ðŸ‘‰ Saga ensures rollback on failure.
ðŸ‘‰ Circuit Breaker prevents repeated calls to failing services.

--------------------------------------------
server:
  port: 5555
  
spring:
  application:
    name: API-Gateway

    gateway:
      routes:
        - id: microservice-api-1
          uri: lb://MICROSERVICES-1
          predicates:
            - Path=/micro1/**
          filters:
            - RewritePath=/micro1/(?<segment>.*), /${segment}
        
        - id: microservice-api-3
          uri: lb://MICROSERVICES-3
          predicates:
            - Path=/micro3/**
          filters:
            - RewritePath=/micro3/(?<segment>.*), /${segment}
